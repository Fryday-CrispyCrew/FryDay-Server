== 사용자(User)

=== 소셜 로그인 (카카오, 네이버)
`POST /api/users/social/login`

카카오, 네이버 소셜 로그인을 처리합니다. Access Token 기반 인증을 사용합니다.

==== 온보딩 상태 (OnboardingStatus)

응답의 `onboardingStatus` 필드는 사용자가 다음에 어떤 화면으로 이동해야 하는지를 나타냅니다.

[cols="1,3,2"]
|===
|상태 |설명 |다음 화면

|`NEEDS_AGREEMENT`
|약관 동의가 필요한 상태 (최초 가입 첫 단계)
|약관 동의 화면

|`NEEDS_NICKNAME`
|닉네임 설정이 필요한 상태 (두 번째 단계)
|닉네임 설정 화면

|`NEEDS_ONBOARDING`
|온보딩 화면 확인이 필요한 상태 (세 번째 단계)
|온보딩 화면

|`COMPLETED`
|모든 설정이 완료된 상태 (정상 로그인)
|메인 화면
|===

NOTE: 온보딩은 NEEDS_AGREEMENT → NEEDS_NICKNAME → NEEDS_ONBOARDING → COMPLETED 순서로 진행됩니다.

NOTE: `instanceId`는 `GET /api/todos` 응답의 `data[].id` 필드입니다. 반복 투두도 날짜별로 독립된 인스턴스 ID를 가집니다.

NOTE: 기존 `PATCH /api/todos/recurrence/{recurrenceId}` API는 단순 규칙 전체 교체만 지원합니다. 이번 API는 scope를 지정해 "이번만 / 이후 전체 / 전체" 중 하나로 세밀하게 수정할 수 있습니다. 반복 투두 수정은 이 API를 사용하세요.

==== scope별 동작

[cols="1,2,3"]
|===
|scope |UI 문구 예시 |동작

|`THIS`
|이번 일정만 수정
|해당 인스턴스에만 override 값이 저장됩니다. 반복 규칙과 다른 인스턴스는 변경되지 않습니다.

|`THIS_AND_FUTURE`
|이번 일정 이후 모든 반복 일정 수정
|선택한 날짜부터 반복 규칙이 분기됩니다. 기존 규칙은 선택한 날짜 하루 전에 종료되고, 새로운 규칙이 해당 날짜부터 시작됩니다.

|`ALL`
|전체 반복 일정 수정
|반복 규칙 자체를 수정합니다. 개별적으로 수정된 인스턴스(`isOverridden=true`)는 override 값이 유지됩니다.
|===

==== payload 필드 구성

`payload`는 **content 필드**와 **rule 필드**로 나뉩니다. 모두 선택사항이며, 변경하고 싶은 필드만 포함하면 됩니다.

[cols="1,1,1,3"]
|===
|필드 |구분 |THIS에서 |설명

|`title`
|content
|적용
|제목

|`memo`
|content
|적용
|메모

|`isAlarmEnabled`
|content
|적용
|알람 활성화 여부

|`alarmTime`
|content
|적용
|알람 시간 (HH:mm:ss)

|`type`
|rule
|**무시됨**
|반복 주기 (`DAILY` / `WEEKLY` / `MONTHLY` / `YEARLY`)

|`frequencyValues`
|rule
|**무시됨**
|반복 상세 값. WEEKLY → `["MONDAY", "WEDNESDAY"]`, MONTHLY/YEARLY → `["15"]` 형식

|`startDate`
|rule
|**무시됨**
|반복 시작일 (YYYY-MM-DD). `scope=ALL` 에서만 유효

|`endDate`
|rule
|**무시됨**
|반복 종료일 (YYYY-MM-DD). 생략하거나 필드 자체를 넣지 않으면 기존 종료 조건을 유지합니다.
|===

NOTE: `scope=THIS`에서 rule 필드(`type`, `frequencyValues`, `startDate`, `endDate`)는 서버에서 무시됩니다. +
`scope=ALL`에서 `startDate`를 변경하면 오늘 이후 인스턴스가 삭제되고 새 규칙으로 재생성됩니다.
